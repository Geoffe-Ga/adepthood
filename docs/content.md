# Editing Course Content

All course content — stage-locked chapters and the free site resources —
lives in the [`Geoffe-Ga/aptitude-course`](https://github.com/Geoffe-Ga/aptitude-course)
repo and is **vendored** into this one at `backend/content/` by
`scripts/sync_content.py`. The app never fetches content over the
network at runtime; the deploy image carries everything it serves.
Design rationale: ADR 0001 (`docs/adr/0001-git-content-pipeline.md`).

## Where things live

| What | File |
| ---- | ---- |
| Vendored content (manifest + Markdown, all 10 stages) | `backend/content/` (synced; do not hand-edit) |
| The manifest contract (JSON Schema, frozen at 1.0.0) | `backend/content/manifest.schema.json` |
| Vendoring audit trail (pinned SHA + tree digest) | `backend/content/CONTENT_VERSION` (generated) |
| Sync command | `backend/scripts/sync_content.py` / `make sync-content REF=<sha>` |
| Drift gate for CI | `make sync-content-check` |
| Manifest reader (validation, indexes, body reads) | `backend/src/services/content_repository.py` |
| Manifest → seeder bridge | `backend/src/content_config.py` |
| Database seeder (`StageContent` rows) | `backend/src/seed_content.py` |
| API endpoints | `backend/src/routers/course.py` |
| In-app reader (native Markdown) | `frontend/src/features/Course/ChapterReader.tsx` |
| Always-available resource chips | `frontend/src/features/Course/SiteResourcesPanel.tsx` |

## The manifest

`backend/content/manifest.json` is generated by the content repo from
each Markdown file's YAML frontmatter and is the single index the app
reads. It is validated against `manifest.schema.json` (a frozen
contract — changing it requires a `schema_version` bump and an ADR
note) every time the app constructs its `ContentRepository` and every
time the sync script runs.

Two collections:

* `chapters[]` — stage-locked, drip-fed reading. Each entry carries
  `id`, `stage` (1–10, identical numbering in both repos), `chapter`,
  `slug`, `title`, `content_type` (`chapter|essay|prompt|video`),
  `release_day`, `order`, and `path` (the Markdown file).
* `site_resources[]` — free evergreen pages (philosophy, about, …),
  each with `slug`, `title`, `description`, `path`. Served
  authenticated but never stage-gated.

## How releases are timed

`release_day` comes straight from the manifest: day `N` means the
chapter drips open `N` days after the user starts the stage (`0` =
immediately). Days at the end of a stage with no new chapter are a
catch-up window by design. Display order within a stage is `order`
(which may interleave essays/prompts between chapters), with ties
broken by `chapter` then `id`.

## Adding, editing, reordering, or removing content

Everything happens in the **content repo**, then gets vendored here:

1. Edit the Markdown / frontmatter in `aptitude-course` and merge.
   The content repo regenerates `manifest.json`.
2. Vendor the new pin from this repo's root:

   ```bash
   make sync-content REF=<commit-sha>
   ```

   The script resolves the ref to a concrete SHA, downloads the GitHub
   tarball (no git binary, no token — the content repo is public),
   validates the manifest against the schema, and atomically swaps the
   published surface into `backend/content/`, stamping
   `CONTENT_VERSION` with the SHA, a timestamp, and a tree digest.
   Always pin a SHA — the script warns if you vendor a mutable ref.
3. Commit the `backend/content/` diff (it is `linguist-vendored`, so
   it stays out of language stats) and deploy.
4. On boot the seeder reconciles `StageContent` rows from the manifest
   for every stage it ships: new chapters insert, changed fields update
   in place, and rows that were marked read stay marked read.

The seeder never deletes rows (deletion would orphan
`ContentCompletion` read-marks). A chapter dropped from the manifest
simply stops being referenced; write a one-off migration if a row must
actually go. Stages the manifest does not cover yet keep placeholder
rows (`seed_content.py`); those suppress automatically once the
manifest ships the stage.

## How the app serves it

* `StageContent.url` holds a local `content://<chapter-id>` reference —
  not a fetchable URL. The body endpoint
  (`GET /course/content/{id}/body`) resolves it through
  `ContentRepository.read_body` after the drip-feed gate, and returns
  raw Markdown (`body_markdown` + `title` + `content_type`); the
  client renders it natively. Locked, unreleased, or unknown content
  is indistinguishable from missing (`404 content_not_found`).
* `GET /course/site-resources` lists the manifest's `site_resources[]`
  (auth required, no stage gating);
  `GET /course/site-resources/{slug}/body` serves a resource body the
  same way chapters are served.
* A manifest-listed file missing from disk is a server bug and returns
  `502 content_unavailable` — there is no auth failure mode because
  local files need no credentials.

## CI drift gate

`make sync-content-check` recomputes the vendored tree's digest and
compares it to `CONTENT_VERSION` without mutating anything — wire-up
into the build pipeline is tracked in the cms-migration epic (#388,
issue #397). A tampered or hand-edited `backend/content/` fails the
check; re-run the sync to fix.

## Local development

No env vars and no network are needed for content. The backend reads
`backend/content/` directly (override the directory with
`CONTENT_DIR` if you vendor content elsewhere). Until a content pin
has been vendored, the app boots cleanly with zero chapters and an
empty resources panel — sync a pin to see real content.
